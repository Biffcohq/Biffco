# ADR-001: Arquitectura Base y Stack Tecnológico (Fase 0)

**Fecha:** Marzo 2026
**Estatus:** Aceptado
**Autor:** Core Team

## Contexto
Durante la Fase 0 (Foundation) se deba estabilizar todo el esqueleto multi-tenant y agnóstico de la plataforma BIFFCO™ enfocada en cadenas de suministro mediante criptografía inmutable. Requeríamos elegir tecnologías subyacentes que garantizaran alto performance, un modelo mental simple para desarrolladores y escalabilidad infraestructural.

## Decisiones

### 1. Sistema de Construcción y Monorepo
Se optó por **pnpm workspaces** en conjunción con **TurboRepo**.
- **Por qué pnpm:** Permite gestionar una única versión de Node y deduplicar el peso de las librerías a través de simlinks virtuales sin sobrecargar a los developers.
- **Por qué TurboRepo:** Minimiza los flujos de CI (Gitleaks, ESLint v9 flat config) al cachear artefactos compilados por hash. Sólo construye el `package` modificado.

### 2. Infraestructura de Computación y Backend
- **Core HTTP:** Construido en Fastify debido a su bajísimo overhead por request en entornos de microservicios o contenedores Edge.
- **Aplicaciones Cliente:** Se implementan **Next.js 14**. Todas las integraciones front usaran SSR y SSR Stream.
- **Estilos:** Se definió un Design System privativo sobre **Tailwind v3.4**, reteniendo la estabilidad clásica contra las betas inestables de la V4 y garantizando un control granular vía tokens semánticos (referencia local `DesignSystem_v1.0`).

### 3. Persistencia de Datos
- **ORMs:** Se abandona Prisma a favor de **Drizzle ORM** debido a que elimina el intermediario del Prisma Client Engine en Rust y corre puramente en V8 sin latencia agregada fundamental para Edge Functions.
- **RDBMS:** Relacional sobre **Neon Serverless Postgres**, lo cual brinda separación de cómputo y storage, pooling nativo y ramificación de Database Branches (útil en dev).
- **In-memory/Cache:** **Upstash Redis**, compatible 100% serverless mediante HTTP REST.

### 4. Seguridad y Gestión de Secretos
- Prohibición sistemática de archivos `.env.local` subidos a GIT por error mediante mitigación continua con **Gitleaks** en CI Action.
- Las variables se proveen *on-the-fly* en las máquinas de desarrolladores vía la CLI de **Doppler**, creando un entorno inalterable entre máquinas, staging y producción.

### 5. Estructura de DB
Se introdujeron patrones de indexación forzada (B-Tree sobre `workspace_id`) y el modelo temporal (Event Sourcing local) se sostiene bajo indexaciones de la variable `stream_id` (AssetID).

## Consecuencias
Esta base agiliza la inducción para el Phase Gate 0 pero acarrea consigo un requerimiento de DevOps inicial para levantar la máquina en Docker Desktop que simula todo el ecosistema local si el developer elige no enlazar el cloud Doppler temporal (vía `setup.sh`).

Las fases subsecuentes solo operarán bajo estos lineamientos fijos.
